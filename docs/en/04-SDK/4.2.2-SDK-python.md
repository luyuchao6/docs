# 4.2.2 SDK-python

## Applicable Versions and System Notes

SDK-python is a general-purpose software interface package developed for Hightorque Electromechanical motor controllers. It provides standardized control and communication capabilities based on CAN and serial communication. The toolkit includes a complete C++ underlying library and corresponding Python bindings, supporting flexible integration into various control system environments. It has the following core features:

- Low-level hardware communication protocol encapsulation
- Multi-modal motor control (position / velocity / torque / composite control)
- Real-time status monitoring and feedback
- Hardware configuration parameter management
- Supports CAN-FD and CAN 2.0B communication protocols
- Applicable versions:
    - SDK program: `v3.0.0+`
    - Motor firmware: `v3.0.0+`
    - Supported systems: `Linux (Ubuntu 20.04 recommended)`
    - Python version: 3.7 and above

### Simple Usage Example

```python
import hightorque_robot_py as hr

# Create a robot instance
robot = hr.Robot()
robot.lcm_enable()  # Enable LCM communication (optional)

# Get motor list
print(f"Found {len(robot.Motors)} motors")

# Control motors
for motor in robot.Motors:
    # Position-velocity-max torque control
    motor.pos_vel_MAXtqe(position=1.0, velocity=0.5, max_torque=10.0)

# Send control commands
robot.motor_send_2()

# Read motor status
for motor in robot.Motors:
    state = motor.get_current_motor_state()
    print(f"Motor {state.ID}: position={state.position:.3f}, velocity={state.velocity:.3f}")

# Stop all motors
robot.set_stop()
robot.motor_send_2()
```

This example is the basic usage framework:

- Create a robot object
- Enable LCM communication (optional)
- Control motors (write motor control commands to the buffer)
- Send control commands
- Read motor status (read data from the buffer; motor status parsing is performed in a sub-thread)
- Stop all motors

## Control Functions

**Term Definitions:**

- Motor status data: includes position, velocity, torque, mode, and error code.
    - Mode and error code require SDK, communication board, and motor firmware versions all above v4; otherwise they are always 0.

### motor

- **Notes:**
    - Unified control: Each operation controls all motors on the same CAN channel.
    - Motors on the same CAN channel must use the same control mode:
        - All motors on the same CAN channel can only use the same control mode.
        - When calling functions of different modes, switching modes will clear the buffer.
    - Batch send: `motor_send_2()` sends all buffered commands at once.
- **Examples:**
    - Example 1:

        Control motors 1 and 2 in position mode with position set to 1, then control motor 3 in velocity mode with velocity set to 0.1, then use the send function `motor_send_2()`. Only motor 3 responds to control; the data for motors 1 and 2 will be cleared.

```python
# Set motors 1 and 2 to position mode
robot.Motors[0].position(1)  # Add to buffer
robot.Motors[1].position(1)  # Add to buffer

# Set motor 3 to velocity mode (mode switch: buffer is cleared first, then command is written)
robot.Motors[2].velocity(0.1)

# Send commands (only velocity mode commands exist at this point)
robot.motor_send_2()  # Only motor 3 will respond; commands for motors 1 and 2 have been cleared
```

    - Example 2:

        The following motor control functions store the control command data frames in an array, which are sent together when the `motor_send_2()` function is called.

```python
# Write control commands to the buffer
robot.Motors[0].velocity(position)
robot.Motors[1].velocity(position)
robot.Motors[2].velocity(position)
robot.Motors[3].velocity(position)
robot.Motors[4].velocity(position)

# Send the buffered control commands
robot.motor_send_2()
```

#### Position Control Mode

**Description:**

1. The motor moves to the specified target position at maximum speed and maximum acceleration, and returns motor status information.
2. Parameter description:
    - `position`: Target position, unit: radians (rad).

**Notes:**

1. In this mode, both speed and torque are at maximum; the motion process is relatively aggressive.
2. Instantaneous current may spike to 5A~10A. If the power supply response is not fast enough, or if the power supply current limit is too small, the motor may not receive sufficient current instantaneously, resulting in an error.
3. This mode is suitable for scenarios with extreme response speed requirements and is generally not recommended. For position control, it is recommended to use the **trapezoidal control** mode.
4. This function only writes the command to the buffer and does not send immediately; `motor_send_2()` must be called to take effect.

```python
motor.position(position)
```

**Notes:**

1. This function only writes the command to the buffer and does not send immediately; `motor_send_2()` must be called to take effect.

#### Position + Velocity + Max Torque Control Mode

**Description:**

1. The motor moves to the specified target position at the target velocity, while limiting the maximum output torque, and returns motor status information.
2. Parameter description:
    - `position`: Target position, unit: radians (rad).
    - `velocity`: Target velocity, unit: radians per second (rad/s).
    - `torque`: Maximum allowed output torque, unit: Newton-meters (N·m).

**Notes:**

1. This mode limits the output torque. If the maximum torque set is too small, the motor may not be able to reach the target velocity.
2. This function only writes the command to the buffer and does not send immediately; `motor_send_2()` must be called to take effect.

```python
motor.pos_vel_MAXtqe(position, velocity, torque)
```

#### Motion Control Mode (No Longer Used; Retained for Compatibility with Legacy Algorithms — Use Motion Control Mode 2 Instead)

**Description:**

1. The motor output torque is calculated as:
    - Output torque = position error * kp + velocity error * kd + feedforward torque
2. Parameter description:
    - `position`: Target position, unit: radians (rad).
    - `velocity`: Target velocity, unit: radians per second (rad/s).
    - `torque`: Feedforward torque, unit: Newton-meters (N·m).

**Notes:**

1. Appropriate `kp` and `kd` values must be set, otherwise the control effect may be poor.
2. Position is obtained by continuous integration, so unexpected situations may occur under low-frequency control. **This function is not recommended**; use motion control mode 2 `pos_vel_tqe_kp_kd2` instead.
3. This function only writes the command to the buffer and does not send immediately; `motor_send_2()` must be called to take effect.

```python
motor.pos_vel_tqe_kp_kd(position, velocity, torque, kp, kd)
```

#### Motion Control Mode 2 (MIT Mode)

**Description:**

1. The motor output torque is calculated as:

```text
Output torque = position error * kp + velocity error * kd + feedforward torque
```

2. Mode usage description:
    1. When kp=0 and kd=0, providing tqe achieves a given torque output. In this case, the motor continuously outputs a constant torque.
    2. When kp=0 and kd≠0, providing vel achieves constant-speed rotation. There is a steady-state velocity error during constant-speed rotation; additionally, kd should not be too large, as an excessively large kd will cause oscillation.
    3. When kp≠0 and kd=0, oscillation will occur. That is, when controlling position, kd must not be set to 0, otherwise the motor will oscillate or even lose control.
    4. When kp≠0 and kd≠0, there are multiple scenarios; two simple cases are described below:
        - When the target position pos is a constant and the target velocity vel is 0, fixed-point control is achieved: the system drives the actual position toward pos and the actual velocity toward 0, eventually coming to rest at the target position.
        - When pos is a continuously differentiable function of time and vel is the derivative of pos, position tracking and velocity tracking are achieved, i.e., rotating the desired angle at the desired velocity.
3. Parameter description:
    - `position`: Target position, unit: radians (rad).
    - `velocity`: Target velocity, unit: radians per second (rad/s).
    - `torque`: Feedforward torque, unit: Newton-meters (N·m).

**Notes:**

1. This is the true motion control mode. The difference from `pos_vel_tqe_kp_kd` is:
    - `pos_vel_tqe_kp_kd` position is obtained by continuous integration, which is prone to unexpected behavior under low-frequency control.
    - `pos_vel_tqe_kp_kd2` avoids this problem and provides more stable control.
2. Appropriate `kp` and `kd` values must be set, otherwise the control effect may be poor.
3. This function only writes the command to the buffer and does not send immediately; `motor_send_2()` must be called to take effect.

```python
motor.pos_vel_tqe_kp_kd2(position, velocity, torque, kp, kd)
```

#### Position + Velocity + PD Control Mode

**Description:**

1. The motor output torque is calculated as:
    - Output torque = (target position - current position) * kp + (target velocity - current velocity) * kd
2. Parameter description:
    - `position`: Target position, unit: radians (rad).
    - `velocity`: Target velocity, unit: radians per second (rad/s).

**Notes:**

1. Appropriate `kp` and `kd` values must be set, otherwise the control effect may be poor.
2. This function only writes the command to the buffer and does not send immediately; `motor_send_2()` must be called to take effect.

```python
motor.pos_vel_kp_kd(position, velocity, kp, kd)
```

#### Position + Velocity + Acceleration Control (Trapezoidal Control) Mode

**Description:**

1. The motor moves at a constant acceleration, achieving a trapezoidal velocity profile of accelerate → constant speed → decelerate, while returning status information.
2. Parameter description:
    - `position`: Target position, unit: radians (rad).
    - `velocity`: Target velocity, unit: radians per second (rad/s).
    - `acc`: Acceleration, unit: revolutions per second squared (rps²).

**Notes:**

1. This function only writes the command to the buffer and does not send immediately; `motor_send_2()` must be called to take effect.

```python
motor.pos_vel_acc(position, velocity, acc)
```

### Robot

#### Stop Function

**Description:**

1. All motors enter stop mode; the three phases of each motor are left floating, allowing the motors to rotate freely.

**Notes:**

1. **Does not return motor status.**
2. When this function is called, the command is sent immediately; there is no need to additionally call the send function `motor_send_2()`.
3. Overwrites existing data in the buffer.

```python
robot.set_stop()
```

#### Brake Function

**Description:**

1. Short-circuits all motor phases to ground, achieving a "damping brake" effect.
2. Braking resistance is positively correlated with motor speed.

**Notes:**

1. **Does not return motor status.**
2. Braking relies on the motor's own damping; the motor can still rotate slowly under external force.
3. When this function is called, the command is sent immediately; there is no need to additionally call the send function `motor_send_2()`.
4. Overwrites existing data in the buffer.

```python
robot.brake()
```

#### Reset Function

**Description:**

1. Performs a software reset on the motors.

**Notes:**

1. **Does not return motor status.**
2. After the reset command is issued, the motor requires approximately 50ms before it can be normally controlled; the function internally delays 100ms.
3. The motor uses a single-turn absolute encoder. After a reset, the read motor position is single-turn data, i.e., within -1.717 to 1.717 radians.
4. When this function is called, the command is sent immediately; there is no need to additionally call the send function `motor_send_2()`.

```python
robot.reset()
```

#### Read Status Function

**Description:**

1. Sends a command to read the current status of the motors, used to read motor status information without controlling the motors.

**Notes:**

1. When this function is called, the command is sent immediately; there is no need to additionally call the send function `motor_send_2()`.
2. Calling this function does not control the motor; it only retrieves status information.

```python
robot.send_get_motor_state_cmd()
```

#### Send Function

**Description:**

1. Sends the command data set in the buffer to the motors.

**Notes:**

1. All motor control functions in `motor` (such as position, velocity, torque, etc.) only write to the buffer and do not take effect immediately. This function must be called to send the commands. Control functions in `robot` internally call `motor_send_2` on their own.

```python
robot.motor_send_2()
```

### **yaml Configuration File Description**

`rb.Motor` is a `vector<motor*>` list, which can be understood as a list containing all motor interfaces on the robot. Motors on the robot can be controlled through this list. The order of this list is determined by the robot's configuration file.

- The SDK provides multiple example robot configuration files; you can create new ones or modify existing ones as needed.
- Using a single motor as an example, the configuration file is written as follows:

```yaml
robot:
  robot_name: "Single_Motor"
  Serial_Type: "/dev/ttyACM"
  Seial_baudrate: 4000000
  motor_timeout_ms: 0  # Set the motor timeout (unit: milliseconds), range: [0, 32760]. If the motor does not receive a new control command within the specified time, it will enter damping mode. Setting this to 0 disables the timeout feature.
  canport_error_output_flag: false  # Communication board CAN channel error output flag (requires communication board firmware version >= v4.7.0)
  board_special_flag: false  # Special version communication board flag; generally not enabled
  CANboard_num: 1
  CANboard:
    No_1_CANboard:
      CANport_num: 1
      CANport:
        CANport_1:
          serial_id: 1
          motor_num: 1
          motor:
            motor1:
              type: "4538_19"
              id: 1
              name: "L_low_foot"
              num: 1
              pos_limit_enable: false
              pos_upper: 5
              pos_lower: -5
              tor_limit_enable: false
              tor_upper: 5
              tor_lower: -3
```

Based on the file above, the configuration file can be divided into the following 4 parts:

1. Robot root node
    - The configuration file starts with `robot`.
2. Software and hardware parameter description
    - `robot_name: "Single_Motor"`: The robot name is `Single_Motor`; can be customized.
    - `Serial_Type: "/dev/ttyACM"`: The robot communication board serial port name is `/dev/ttyACM*`; generally no change is needed.
    - `Serial_baudrate: 4000000`: The robot communication board serial baud rate is 4000000; generally no change is needed.
    - `motor_timeout_ms: 0`: Sets the timeout:
        - When the motor does not receive a new command within the set time, it enters damping mode, used to prevent the motor from continuing to move when CAN communication is abnormal.
        - Default value of 0 means this feature is disabled.
    - `canport_error_output_flag: false`: Communication board CAN channel error output flag. `true` means enabled, `false` means disabled.
    - `board_special_flag: false`: Special version communication board flag; generally not enabled. `true` means enabled, `false` means disabled.
    - `CANboard_num: 1`: Number of communication boards used by the robot.
3. Communication board configuration information
    - `CANboard:`: Communication board information for the robot, including CAN channel information and motor information for each CAN channel.
    - `No_1_CANboard:`: Node for the first communication board.
    - `CANport_num: 1`: Number of CAN channels used by the communication board.
    - `CANport`: CAN channel information, including CAN channel nodes and motor information on each CAN channel.
    - `CANport_1`: Node for the first CAN channel; it is just a name and must start from `CANport_1`.
    - `serial_id: 1`: The actual CAN channel in use.
    - `motor_num: 1`: Number of motors on this CAN channel.
4. Motor configuration information
    - `motor_1`: Information for the 1st motor on this CAN channel.
    - `type: "4438_30"`: Motor type, used for motor output torque calculation. An incorrect value does not affect motor rotation but affects the accuracy of the output torque. Available options are:
        - `3536_32`
        - `4538_19`
        - `5046_20`
        - `5047_09`
        - `5047_36`: This is the old version of 5047_36, now discontinued. Retained for compatibility with legacy algorithms; use 5047_36_2 instead.
        - `4438_30`
        - `4438_32`
        - `6056_36`
        - `5043_20`
        - `7256_35`
        - `60sg_35`
        - `60bm_35`
        - `5047_36_2`: Equivalent to the new version of 5047_36; recommended for use.
        - `General`: For motors not yet supported by the current SDK, this can be used for approximate correction.
    - `id: 1`: Motor ID
    - `name: "L_low_foot"`: Motor name; can be customized.
    - `pos_limit_enable: false`: Position protection enable flag. `true` means enabled, `false` means disabled.
    - `pos_upper: 5`: Position upper limit, unit: radians (rad).
    - `pos_lower: -5`: Position lower limit, unit: radians (rad).
    - `tor_limit_enable: false`: Torque limit enable flag. `true` means enabled, `false` means disabled.
    - `tor_upper: 5`: Torque upper limit, unit: Newton-meters (N·m).
    - `tor_lower: -3`: Torque lower limit, unit: Newton-meters (N·m).

**Notes:**

    - The motor ID on each CAN channel must start from 1.
    - The `id` in the configuration file must match the ID of the motor in use.
    - `serial_id` must match the actual connected CAN channel.
    - Motors are sorted in the order they appear in the configuration file:
        - First sorted by `CANport` order (`CANport_1` → `CANport_2` → ...)
        - Within each `CANport`, sorted by `motor` order (`motor1` → `motor2` → ...)
        - Example:
            - Using the `6dof_*.yaml` configuration file with 12 motors connected. For simplicity, only the CANport and motor relationships are shown; other detailed configurations are omitted.

```python

    CANport_1:
      motor:
        motor1:...
        motor2:...
        motor3:...
        motor4:...

        motor6:...

      motor:
        motor1:...
        motor2:...
        motor3:...
        motor4:...
        motor5:...
        motor6:...
```

            - To control `motor1` and `motor2` of `CANport_1`, and `motor2` and `motor3` of `CANport_2`, the operations are as follows:

## Frequently Asked Questions

1. Is `state.torque` the total torque, external torque, or internal torque?

Answer: This is the motor output torque.
