# 1.2 FDCAN Protocol Reference

**It is recommended to directly port and use the example routines, see** [03-Examples](../03-motor-usage-examples/)

---

### Protocol Reference

#### CAN-FD Related Notes

1. CAN-FD baud rate:
    - Arbitration segment: 1 Mbps
    - Data segment: up to 5 Mbps, or 1 Mbps can also be used (variable baud rate).
2. ID: composed of 16 bits, where `0x7F` is the broadcast address.
    - High 8 bits: represent the **source address**:
        - Most significant bit = 1: reply required; acts as a **master switch** — enabling it without issuing a query command will return a frame with a data length of 0.
        - Most significant bit = 0: no reply required.
        - Remaining 7 bits: source address.
    - Low 8 bits: represent the **destination address**:
        - Most significant bit = 0.
        - Remaining 7 bits represent the destination address.

        **Examples:**

        - `ID: 0x8001`
            - Source address is 0.
            - Destination address is 1.
            - Most significant bit is 1, indicating a reply is required — the reply **master switch** is on.
        - `ID: 0x100`
            - Source address is 1.
            - Destination address is 0.
            - Most significant bit is 0, indicating no reply is required — the reply **master switch** is off.
3. Extended frames and standard frames
    - Host → Motor (command frame)
        - Sending control commands (e.g., speed, position control) to a motor with any ID: **both extended and standard frames can be used**.
        - Reading motor status requires an **extended frame**.
    - Motor → Host (feedback frame)
        - Motors with ID 1–7 report status using **standard frames**.
        - Motors with ID 8 and above report status using **extended frames**.

#### Basic Protocol Description

1. The minimum unit is a sub-frame; each sub-frame can write a value to **one or more** registers, or read data.
2. One CAN-FD frame can consist of **one or more** sub-frames.
3. Each motor function is implemented by writing the values of **one or more** corresponding registers into a single FDCAN frame.
4. Four basic data types can be written to any register: `int8_t`, `int16_t`, `int32_t`, `float`.
5. Data types within the **same sub-frame** must be identical; data types across **different sub-frames** may differ.
6. All basic type transfers use **little-endian** format, i.e., the low byte is sent first, followed by the high byte.
7. Padding bytes at the **end** of a CAN-FD frame should use `0x50`, which represents no operation (NOP).
8. **Unlimited** values for each data type:
    - `int8_t`: `0x80`
    - `int16_t`: `0x8000`
    - `int32_t`: `0x80000000`
    - `float`: `NAN`
9. Mode 3 (one-to-many mode) is identified by the FDCAN ID.
    - When the low 8 bits of the ID are greater than or equal to `0x80`, it is Mode 3 (one-to-many mode).
    - When the low 8 bits of the ID are less than `0x80`, it is Mode 1 or Mode 2 (normal mode).

#### Sub-frame Reference

##### Transmit Protocol

###### Mode 1

```c
uint8_t tdata[] = {cmd, addr, a1, a2, b1, b2...};
```

1. `cmd`: indicates read/write, data type, and quantity.
    1. High nibble `cmd[7:4]` indicates read, write, or reply.
        1. `0000`: write
        2. `0001`: read
        3. `0010`: reply
    2. Low nibble `cmd[3:0]` indicates data type and quantity:
        1. High 2 bits `cmd[3:2]` indicate data type:
            1. `00`: `int8_t`
            2. `01`: `int16_t`
            3. `10`: `int32_t`
            4. `11`: `float`
        2. Low 2 bits `cmd[1:0]` indicate data count.
            1. `01`: one data item.
            2. `10`: two data items.
            3. `11`: three data items.
            4. `00`: **Mode 2** flag.
2. `addr`: starting register address.
3. `a1, a2, b1, b2...`: data to be written to the registers. Note: must comply with items 4 and 5 of section 1.2 Basic Protocol Description.
    1. `a1, a2`: data to be written to `addr`.
    2. `b1, b2`: data to be written to `addr + 1`.
    3. `...`: data to be written to `addr + n`.

###### Mode 2

```c
uint8_t tdata[] = {cmd, num, addr, a1, a2, b1, b2...};
```

1. `cmd`: indicates read/write, data type, and quantity.
    1. High nibble `cmd[7:4]` indicates read, write, or reply.
        1. `0000`: write
        2. `0001`: read
        3. `0010`: reply
    2. Low nibble `cmd[3:0]` indicates data type and quantity:
        1. High 2 bits `cmd[3:2]` indicate data type:
            1. `00`: `int8_t`
            2. `01`: `int16_t`
            3. `10`: `int32_t`
            4. `11`: `float`
        2. Low 2 bits `cmd[1:0]` are fixed at `00`, indicating Mode 2; the following byte represents the data count.
2. `num`: data count.
3. `addr`: starting register address.
4. `a1, a2, b1, b2...`: data to be written to the registers. Note: must comply with items 4 and 5 of section 1.2 Basic Protocol Description.
    1. `a1, a2`: data to be written to `addr`.
    2. `b1, b2`: data to be written to `addr + 1`.
    3. `...`: data to be written to `addr + n`.

**The essence of Mode 2: when the data count field `xxx` in Mode 1 is set to 0, the following byte is used to represent the data count, and all subsequent bytes are shifted back by one.**

###### Mode 3 (One-to-Many Mode)

- Mode 3 is determined by the **low 8 bits** of the FDCAN ID. When the FDCAN ID is greater than **0x80**, it is **one-to-many mode** (see the example routines for specific modes).
- The protocol format is: Motor 1 data, Motor 2 data, Motor 3 data... The **last 2 bytes** are the motor status query code.
- Motor 1, Motor 2...: refers to the motor with ID 1, the motor with ID 2, etc.
- Data sent in one-to-many mode is of **int16 type**.

**Example 1: ID = 0x8090**

```c
uint8_t cmd[] = {pos11, pos12, val11, val12, tqe11, tqe12, pos21, pos22, val21, val22, tqe21, tqe22, padding_byte, 0x17, 0x01}
```

- When ID is 0x8090, it is one-to-many position, velocity, and torque control.
- Bytes 1 to 6 are position, velocity, and torque control respectively.
- Bytes 7 to 12 are position, velocity, and torque control respectively.
- Bytes 13 to 18 are position, velocity, and torque control respectively.
- And so on...
- The last 2 bytes are the motor status query command.

**Example 2: ID = 0x8093**

```c
uint8_t cmd[] = {pos11, pos12, val11, val12, tqe11, tqe12, rkp11, rkp12, rkd11, rkd12, pos21, pos22, val21, val22, tqe21, tqe22, rkp21, rkp22, rkd21, rkd22, padding_byte, 0x17, 0x01}
```

- When ID is 0x8093, it is one-to-many position, velocity, torque, and PD control.
- Bytes 1 to 10 are position, velocity, torque, and PD control respectively.
- Bytes 11 to 20 are position, velocity, torque, and PD control respectively.
- Bytes 21 to 30 are position, velocity, torque, and PD control respectively.
- And so on...
- The last 2 bytes are the motor status query command.

##### Receive Protocol

- **The receive protocol mode is determined by the transmit protocol mode.**
- **The receive protocol mode and the transmit protocol mode are the same.**

###### Mode 1

Assuming the received data is `int16_t`

```c
uint8_t rdata[] = {cmd, addr, a1, a2, b1, b2, ..., cmd1, addr1, c1, c2, c3, c4}
```

- `cmd`:
    - High nibble `cmd[7, 4]`: `0010` indicates reply.
    - Bits 2–3 `cmd[3, 2]`: indicate type.
        - `00`: `int8_t` type.
        - `01`: `int16_t` type.
        - `10`: `int32_t` type.
        - `11`: `float` type.
    - Low 2 bits `cmd[1, 0]`: indicate count.
        - `00`: indicates Mode 2.
        - `01`: one data item.
        - `10`: two data items.
        - `11`: three data items.
- `addr`: starting address for reading.
- `a1, a2`: data item 1, little-endian.
- `b1, b2`: data item 2, little-endian.

###### Mode 2

```c
uint8_t rdata[] = {cmd, addr, num, a1, a2, b1, b2, ..., cmd1, addr1, c1, c2, c3, c4}
```

- `cmd`:
    - High nibble `cmd[7, 4]`: `0010` indicates reply.
    - Bits 2–3 `cmd[3, 2]`: indicate type.
        - `00`: `int8_t` type.
        - `01`: `int16_t` type.
        - `10`: `int32_t` type.
        - `11`: `float` type.
    - Low 2 bits `cmd[1, 0]`: indicate count.
        - `00`: indicates Mode 2; the following byte represents the data count.
- `num`: data count.
- `addr`: starting address for reading.
- `a1, a2`: data item 1, little-endian.
- `b1, b2`: data item 2, little-endian.

##### Examples

###### Transmit Protocol Examples

###### Mode 1

```c
uint8_t cmd[] = {0x01, 0x00, 0x0A, 0x0A, 0x20, 0x00, 0x00, 0x00, 0x80, 0x10, 0x27, 0x00, 0x00, 0x50, 0x50, 0x50};
```

**This CAN-FD frame consists of two sub-frames:**

1. Sub-frame 1: the overall meaning is that the motor enters position mode.
    - `0x01`: start of the first sub-frame
        - High nibble is `0000`, indicating a write operation.
        - Low nibble:
            - High 2 bits: `00`, indicating `int8_t` type.
            - Low 2 bits: `01`, indicating 1 data item.
    - `0x00`: starting register address: from the register table, register `0x00` represents the motor mode setting.
    - `0x0A`: write `0x0A` to register `0x00`.
2. Sub-frame 2: the overall meaning is position unlimited, speed = 0.1 rev/s.
    - `0x0A`: start of the 2nd sub-frame
        - High nibble is `0000`, indicating a write operation.
        - Low nibble:
            - High 2 bits: `10`, indicating `int32_t` type.
            - Low 2 bits: `10`, indicating 2 data items.
    - `0x20`: starting register address: from the register table, register `0x20` represents position, register `0x21` represents speed.
    - `0x00, 0x00, 0x00, 0x80`: little-endian, i.e., `0x80000000` is written to register `0x20`, meaning motor position is unlimited.
    - `0x10, 0x27, 0x00, 0x00`: little-endian, i.e., `0x2710` is written to register `0x21`, meaning motor speed is set to 0.1 rev/s.
3. `0x50, 0x50, 0x50`: since the nearest valid CAN-FD byte counts are 12 and 16, 3 padding bytes are needed.

###### Mode 2

```c
uint8_t cmd[] = {0x01, 0x00, 0x0A, 0x08, 0x02, 0x20, 0x00, 0x00, 0x00, 0x80, 0x10, 0x27, 0x00, 0x00, 0x50, 0x50};
```

**This CAN-FD frame consists of two sub-frames:**

1. Sub-frame 1: the overall meaning is that the motor enters position mode.
    - `0x01`: start of the first sub-frame
        - High nibble is `0000`, indicating a write operation.
        - Low nibble:
            - High 2 bits: `00`, indicating `int8_t` type.
            - Low 2 bits: `01`, indicating 1 data item.
    - `0x00`: starting register address: from the register table, register `0x00` represents the motor mode setting.
    - `0x0A`: write `0x0A` to register `0x00`.
2. Sub-frame 2: the overall meaning is position unlimited, speed = 0.1 rev/s.
    - `0x08`: start of the 2nd sub-frame
        - High nibble is `0000`, indicating a write operation.
        - Low nibble:
            - High 2 bits: `10`, indicating `int32_t` type.
            - Low 2 bits: `00`, indicating Mode 2; the following byte represents the data count.
    - `0x02`: 2 data items.
    - `0x20`: starting register address: from the register table, register `0x20` represents position, register `0x21` represents speed.
    - `0x00, 0x00, 0x00, 0x80`: little-endian, i.e., `0x80000000` is written to register `0x20`, meaning motor position is unlimited.
    - `0x10, 0x27, 0x00, 0x00`: little-endian, i.e., `0x2710` is written to register `0x21`, meaning motor speed is set to 0.1 rev/s.
3. `0x50, 0x50`: since the nearest valid CAN-FD byte counts are 12 and 16, 2 padding bytes are needed.

###### Receive Protocol Examples

- **The receive protocol mode is determined by the transmit protocol mode.**
- **The receive protocol mode and the transmit protocol mode are the same.**

###### Mode 1

```c
uint8_t rdata[] = {0x2B, 0x01, 0x96, 0x1D, 0x04, 0x00, 0x54, 0x40, 0x02, 0x00, 0x86, 0x01, 0x00, 0x00, 0x50, 0x50};
```

**This frame consists of one sub-frame:**

1. Sub-frame 1: the overall meaning is that the motor is in position mode.
    - `0x2B`:
        - High nibble is `0010`, indicating a reply operation.
        - Low nibble:
            - High 2 bits: `10`, indicating `int32_t` type.
            - Low 2 bits: `11`, indicating 3 data items.
    - `0x01`: starting register address.
    - `0x96, 0x1D, 0x04, 0x00`: little-endian, decimal value 269718, meaning the motor's current position is 2.69718 revolutions.
    - `0x54, 0x40, 0x02, 0x00`: little-endian, decimal value 147540, meaning the current motor speed is 1.4754 rev/s.
    - `0x86, 0x01, 0x00, 0x00`: little-endian, decimal value 390, meaning the current actual output torque is 0.0039 Nm.
    - `0x50, 0x50`: padding bytes.

**From the first three bytes:** this CAN-FD frame is a reply to `0x1B, 0x01`.

###### Mode 2

```c
uint8_t rdata[] = {0x28, 0x04, 0x00, 0x0A, 0x00, 0x00, 0x00, 0x96, 0x1D, 0x04, 0x00, 0x54, 0x40, 0x02, 0x00, 0x86, 0x01, 0x00, 0x00, 0x50};
```

**This frame consists of one sub-frame:**

1. Sub-frame 1: the overall meaning is that the motor is in position mode.
    - `0x28`:
        - High nibble is `0010`, indicating a reply operation.
        - Low nibble:
            - High 2 bits: `10`, indicating `int32_t` type.
            - Low 2 bits: `00`, indicating Mode 2; the following byte represents the data count.
    - `0x04`: 4 data items.
    - `0x00`: starting register address.
    - `0x0A, 0x00, 0x00, 0x00`: value of register `0x00`; from the register table, this is the motor mode; decimal value 10 corresponds to position mode.
    - `0x96, 0x1D, 0x04, 0x00`: little-endian, decimal value 269718, meaning the motor's current position is 2.69718 revolutions.
    - `0x54, 0x40, 0x02, 0x00`: little-endian, decimal value 147540, meaning the current motor speed is 1.4754 rev/s.
    - `0x86, 0x01, 0x00, 0x00`: little-endian, decimal value 390, meaning the current actual output torque is 0.0039 Nm.
    - `0x50`: padding byte.

**From the first three bytes:** this CAN-FD frame is a reply to `0x18, 0x04, 0x00`.

##### Position Mode

**Description:**

1. The motor moves to the specified target position at maximum speed and maximum acceleration, and causes the motor to return status information.
2. Low-level implementation:

```c
void set_pos_float(FDCAN_HandleTypeDef *fdcanHandle, uint8_t id, float pos)；
void set_pos_int16(FDCAN_HandleTypeDef *fdcanHandle, uint8_t id, int16_t pos)；
void set_pos_int32(FDCAN_HandleTypeDef *fdcanHandle, uint8_t id, int32_t pos)；
```

**Notes:**

1. In this mode, both speed and torque are at maximum, resulting in aggressive motion.
2. Instantaneous current may spike to 5A–10A. If the power supply response is too slow or the current limit is too low, the motor may not receive sufficient current instantaneously, causing an error.
3. This mode is intended for applications with extreme response speed requirements and is generally not recommended. For position control, the **trapezoidal control** mode is recommended.

##### Motion Control Mode (MIT Control)

**Description:**

1. The motor output torque is calculated as:

```text
Output torque = position error * kp + velocity error * kd + torque.
```

2. Mode usage notes:
    1. When kp=0, kd=0: providing a tqe value achieves a given torque output. In this case, the motor continuously outputs a constant torque.
    2. When kp=0, kd≠0: providing a vel value achieves constant speed rotation. There is a steady-state speed error during constant speed rotation; additionally, kd should not be set too high, as excessive kd will cause oscillation.
    3. When kp≠0, kd=0: oscillation will occur. When controlling position, kd must not be set to 0, otherwise the motor will oscillate or even lose control.
    4. When kp≠0, kd≠0: there are multiple scenarios; two simple cases are described below.
        - When target position pos is constant and target velocity vel is 0, point control is achieved: the system drives the actual position toward pos and the actual velocity toward 0, eventually coming to rest at the target position.
        - When pos is a continuously differentiable function of time and vel is the derivative of pos, both position tracking and velocity tracking are achieved, i.e., the motor rotates the desired angle at the desired speed.
3. Low-level implementation:

```c
void set_pos_vel_tqe_kp_kd_float_2(FDCAN_HandleTypeDef *fdcanHandle, uint8_t id, float pos, float vel, float tqe, float kp, float kd)；
void set_pos_vel_tqe_kp_kd_int16_2(FDCAN_HandleTypeDef *fdcanHandle, uint8_t id, int16_t pos, int16_t vel, int16_t tqe, int16_t kp, int16_t kd)；

```

**Notes:**

1. This is the true motion control mode. The difference from `motor_set_pos_vel_tqe_kp_kd` is:
    - `motor_set_pos_vel_tqe_kp_kd`: position is obtained by continuous integration, which can lead to unexpected behavior under low-frequency control.
    - `motor_set_pos_vel_tqe_kp_kd_2`: avoids this problem, providing more stable control.
2. Appropriate `kp` and `kd` values must be set, otherwise control performance may be poor.


#### One-to-Many Control Mode `motor_many.c`

- In this file, only `motor_many_send` is used to send commands; all other functions write commands to a buffer.
- In one-to-many mode, all motors on the same CAN channel operate in the same mode.
- The general principle of one-to-many mode: a single generic FDCAN frame is sent, where different bytes of the frame control different motors; the motor mode is determined by the ID. For details, see the FDCAN protocol reference (section 02).
- In one-to-many mode, with the arbitration segment at 1M and data segment at 5M, a single CAN bus can control up to 10 motors within 1 ms, i.e., up to 10 motors at 1 kHz control frequency. To control more motors, the control frequency must be reduced.
- `TINT16` corresponds to the `int16` data type; position control is limited to 3.2 revolutions.
- One-to-many mode **usage instructions**:

```text
/* Write to buffer */
motor_many_vel(PORT1, 1, 0.1);
motor_many_vel(PORT1, 2, 0.1);
motor_many_vel(PORT1, 3, 0.1);

/* Send command */
motor_many_send(PORT1);
```

**Notes:**
