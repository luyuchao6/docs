# 1.3 CAN Protocol Reference

**It is recommended to directly port and use the example routines, see** [03-Examples](../03-motor-example-code/)

---

## Protocol Reference

### CAN Related Notes

1. CAN baud rate:
    - Arbitration segment: 1 Mbps
    - Data segment: 1 Mbps
2. ID: composed of 16 bits, where 0x7F is the broadcast address.
    - High 8 bits: source address:
        - Highest bit is 1: a reply is required. This acts as a master switch; enabling it without sending a query command will return a frame with a data segment length of 0.
        - Highest bit is 0: no reply required.
        - Remaining 7 bits: source address.
    - Low 8 bits: destination address:
        - Highest bit is 0.
        - Remaining 7 bits: destination address.

        **Examples:**

        - `ID: 0x8001`
            - Source address is 0.
            - Destination address is 1.
            - Highest bit is 1, indicating a reply is required, i.e., the reply **master switch** is on.
        - `ID: 0x100`
            - Source address is 1.
            - Destination address is 0.
            - Highest bit is 0, indicating no reply is required, i.e., the reply **master switch** is off.
3. Extended frames and standard frames
    - Sending
        - **Both can be used** for motor control
        - Reading motor status requires an **extended frame**
    - Motor feedback
        - Motors with IDs 1–7 use **standard frames** for feedback status
        - Motors with IDs 8 and above use **extended frames** for feedback status

### Mode Description

#### Normal Mode (position and velocity cannot be controlled simultaneously)

```xml
uint8_t cmd[] = {0x07, 0x07, pos1, pos2, val1, val2, tqe1, tqe2};
```

The normal protocol consists of: command bytes (2 bytes) + position (2 bytes) + velocity (2 bytes) + torque (2 bytes), totaling 8 bytes.

`0x07 0x07`: Normal mode, which can control velocity and torque, or position and torque (see [[#2.1 Normal Mode]]).

Position, velocity, and torque data in the protocol are all in little-endian format, i.e., the low byte is sent first, followed by the high byte.

For example, for `pos = 0x1234`, `pos1 = 0x34`, `pos2 = 0x12`.

This mode supports two control methods:

Position and torque control (in this case `val=0x8000` means unlimited).

Velocity and torque control (in this case `pos=0x8000` means unlimited).

#### Torque Mode

```xml
uint8_t cmd[] = {0x05, 0x13, tqe1, tqe2};
```

The torque mode protocol consists of: command bytes (2 bytes) + torque (2 bytes).

`0x05 0x13`: Pure torque mode, followed by two bytes of torque data. (See [[#2.3 Torque Mode]]).

Torque data in the protocol is in little-endian format, i.e., the low byte is sent first, followed by the high byte.

For example, for `tqe = 0x1234`, `tqe1 = 0x34`, `tqe2 = 0x12`.

#### Coordinated Control Mode (position, velocity, and torque can be controlled simultaneously)

```xml
uint8_t cmd[] = {0x07, 0x35, val1, val2, tqe1, teq2, pos1, pos2};
```

The coordinated control mode protocol consists of: command bytes (2 bytes) + velocity (2 bytes) + torque (2 bytes) + position (2 bytes), totaling 8 bytes.

`0x07 0x35`: Coordinated control mode, rotates to a specified position at a specified velocity while limiting the maximum torque.

In this mode, passing `0x8000` as a parameter means unlimited (unlimited velocity and torque means maximum value).

For example, `val = 2000`, `tqe = 1000`, `pos = 0x8000`: the motor rotates continuously at 0.5 rev/s with a maximum torque of 0.1 NM.

Position, velocity, and torque data in the protocol are all in little-endian format, i.e., the low byte is sent first, followed by the high byte.

For example, for `pos = 0x1234`, `pos1 = 0x34`, `pos2 = 0x12`.

### Reading Motor Status Data

The protocol for reading motor status is the same as in CAN-FD. The only difference is that CAN is limited to an 8-byte data segment.

For register addresses and function descriptions, refer to the **Register Functions, Motor Operating Modes, and Error Code Description.xlsx** file.

Due to CAN's 8-byte data segment limitation, the motor information that can be returned in a single CAN frame is limited to:

One `float` or `int32_t` type motor register value.

3 consecutive `int16_t` type motor values.

6 consecutive `int8_t` type motor values.

The example routines include sample functions for querying motor position, velocity, and torque using `int16_t`, as well as motor data parsing (the examples use a C language union to directly copy bytes 3 through 8 from the CAN frame).

#### Transmit Protocol Description

```xml
uint8_t tdata[] = {cmd, addr, cmd1, addr1, cmd2, add2};
```

General meaning: read cmd[0, 1] values of type cmd[3, 2] starting from addr.

cmd:

High nibble [7, 4]: 0001 means read.

Bits 2–3 [3, 2]: data type.

00: int8_t type.

01: int16_t type.

10: int32_t type.

11: float type.

Low 2 bits [1, 0]: quantity.

01: one value.

10: two values.

11: three values.

addr: starting address to read from.

Multiple `cmd, addr` pairs can be concatenated to read non-contiguous addresses and different data types in a single request.

#### Receive Protocol Description

Assuming the received data type is `uint16_t`.

```xml
uint8_t rdata[] = {cmd, addr, a1, a2, b1, b2, ..., cmd1, addr1, c1, c2, c3, c4}
```

- `cmd`:
    - High nibble `[7, 4]`: `0010` means reply.
    - Bits 2–3 `[3, 2]`: data type.
        - `00`: `int8_t` type.
        - `01`: `int16_t` type.
        - `10`: `int32_t` type.
        - `11`: `float` type.
    - Low 2 bits `[1, 0]`: quantity.
        - `01`: one value.
        - `10`: two values.
        - `11`: three values.
- `addr`: starting address.
- `a1, a2`: data value 1, little-endian.
- `b1, b2`: data value 2, little-endian.

#### Example

We need to read position, velocity, and torque data.

From the register Excel table, the data addresses for position, velocity, and torque are: 01, 02, and 03 respectively.

Therefore, we can read 3 consecutive values starting from address 01. Given that CAN can transfer at most 8 bytes at a time, and cmd + addr occupy 2 bytes, the maximum data type that can be used is `int16_t`.

From the above, the binary of `cmd` is: `0001 0111`, hexadecimal: `0x17`.

Since we are reading from address `01`, `addr` is `0x01`.

The total data to send is `uint8_t tdata[] = {0x17, 0x01}`.

Example code:

```c
/**
@brief Read motor
@param id
*/
void motor_read(uint8_t id)
{
    static uint8_t tdata[8] = {0x17, 0x01};

    CAN_Send_Msg(0x8000 | id, tdata, sizeof(tdata));
}
uint8_t cmd[] = {0x17, 0x01};
```

The overall meaning is: starting from address `0x01`, read 3 `int16_t` registers (from the table, registers at addresses `0x01~0x03` represent position, velocity, and torque respectively), so this command queries the motor's position, velocity, and torque.

- `0x17`:
    - Binary of `0x17[7:4]` is 0001: means read.
    - Binary of `0x17[3:2]` is 01: data type is int16_t.
    - Binary of `0x17[1:0]` is 11: number of values is 3.
- `0x01`:
    - Start from address `0x01`.

Corresponding received data example:

```xml
uint8_t rdata[] = {0x27, 0x01, 0x38, 0xf6, 0x09, 0x00, 0x00, 0x00};
```

- `0x27`: corresponds to the sent `0x17`.
- `0x01`: starting from address 0x01.
- `0x38 0xf6`: position data: `0xf638`, which is `-2505`.
- `0x09 0x00`: velocity data: `0x0009`, which is `9`.
- `0x00 0x00`: torque data: `0x0000`, which is `0`.

### Motor Stop

Description:

1. Stop the motor.

```xml
/**
@brief Motor stop
*/
void motor_stop(uint8_t id)
{
 uint8_t tdata[] = {0x01, 0x00, 0x00};

 CAN_Send_Msg(0x8000 | id, tdata, sizeof(tdata));
}
```

### Reset Motor Zero Position

Description:

1. Set the current position as the motor zero position.
2. This command only modifies the value in `RAM`; it must be used together with the `conf write` command to save it to `flash`.
3. It is recommended to send the `conf write` command approximately 1 second after issuing this command.

```xml
void rezero_pos(uint8_t id)
{
 uint8_t tdata[] = {0x40, 0x01, 0x04, 0x64, 0x20, 0x63, 0x0a};

 CAN_Send_Msg(0x8000 | id, tdata, sizeof(tdata));
 HAL_Delay(1000);  // Recommended delay of 1s

 conf_write(id);  // Save settings
}
```

### Save Motor Settings (conf write)

Description:

1. Save the motor's RAM settings to flash.
2. It is recommended to power cycle the motor after using this command.

```xml
void conf_write(uint8_t id)
{
 uint8_t tdata[] = {0x05, 0xb3, 0x02, 0x00, 0x00};

 CAN_Send_Msg(0x8000 | id, tdata, sizeof(tdata));
}
```

### Read Motor Status

Description:

1. Read motor position, velocity, and torque data once.
2. For parsing the motor feedback status data, refer to the interrupt function HAL_FDCAN_RxFifo0Callback in the example routines.

```xml
/**
@brief Command to read motor position, velocity, and torque
@param id Motor ID
*/
void motor_read(CAN_HandleTypeDef *hcan, uint8_t id)
{
 static uint8_t tdata[8] = {0x17, 0x01};

 can_send(hcan, 0x8000 | id, tdata, sizeof(tdata));
}
```

### Periodic Motor Status Return

Description:

1. Periodically return motor position, velocity, and torque data.
2. The return data format is the same as that obtained using the `0x17, 0x01` command (i.e., 1.7 Read Position Status).
3. The period unit is ms.
4. The minimum period is 1 ms.
5. To stop periodic data return, set the period to 0 or power off the motor.
6. For parsing the motor feedback status data, refer to the interrupt function HAL_FDCAN_RxFifo0Callback in the example routines.

```xml
void timed_return_motor_status(uint8_t id, int16_t t_ms)
{
 uint8_t tdata[] = {0x05, 0xb4, 0x02, 0x00, 0x00};

 *(int16_t *)&tdata[3] = t_ms;

 CAN_Send_Msg(0x8000 | id, tdata, sizeof(tdata));
}
```

## Example Functions

### DQ Voltage Mode

**Description:**

1. Controls the motor using a given Q-phase voltage and requests the motor to return status information.
2. Data frame

```c
void motor_control_volt(FDCAN_HandleTypeDef *hfdcanx, uint8_t id, int16_t vol)
{
    static uint8_t tdata[] = {0x01, 0x00, 0x08, 0x05, 0x1b, 0x00, 0x00};

    // *(int16_t *)&tdata[5] = vol;
    my_memcpy(&tdata[5], &vol, sizeof(int16_t));

    can_send(hfdcanx, id, tdata, sizeof(tdata));
}
```

### DQ Current Mode

**Description:**

1. Controls the motor using a given Q-phase current and requests the motor to return status information.
2. Data frame

```c
void motor_control_cur(FDCAN_HandleTypeDef *hfdcanx, uint8_t id, int16_t cur)
{
    static uint8_t tdata[] = {0x01, 0x00, 0x09, 0x05, 0x1c, 0x00, 0x00};

    // *(int16_t *)&tdata[5] = cur;
    my_memcpy(&tdata[5], &cur, sizeof(int16_t));

    can_send(hfdcanx, id, tdata, sizeof(tdata));
}
```

### Position Mode

**Description:**

1. The motor moves to the specified target position at maximum velocity and maximum acceleration, and requests the motor to return status information.
2. Underlying implementation:

```c
void motor_control_pos(FDCAN_HandleTypeDef *hfdcanx, uint8_t id, int32_t pos, int16_t tqe)；
```

**Note:**

1. In this mode, velocity and torque are both at maximum, resulting in aggressive motion.
2. Instantaneous current may spike to 5A–10A. If the power supply response is too slow or its current limit is too low, the motor may not receive sufficient current instantaneously, causing an error.
3. This mode is suited for scenarios with extreme response time requirements and is generally not recommended. For position control, the **trapezoidal control** mode is recommended.

### Velocity Mode

**Description:**

1. The motor accelerates to the specified target velocity at maximum acceleration, and requests the motor to return status information.
2. Underlying implementation:

```c
void motor_control_vel(FDCAN_HandleTypeDef *hfdcanx, uint8_t id, int16_t vel, int16_t tqe)；
```

### Torque Mode

**Description:**

1. The motor rotates according to the set target torque, and requests the motor to return status information.
2. Underlying implementation:

```c
void motor_control_tqe(FDCAN_HandleTypeDef *hfdcanx, uint8_t id, int32_t tqe)；
```

### Position and Velocity Mode

**Description:**

1. The motor moves to the specified target position at the target velocity, and requests the motor to return status information.
2. Underlying implementation:

```c
void motor_control_pos_vel_MAXtqe(FDCAN_HandleTypeDef *hfdcanx, uint8_t id, int16_t pos, int16_t vel, int16_t tqe)；
```

### Position, Velocity, and Maximum Torque Mode

**Description:**

1. The motor moves to the specified target position at the target velocity while limiting the maximum output torque, and requests the motor to return status information.
2. Underlying implementation:

```c
void motor_control_pos_vel_MAXtqe(FDCAN_HandleTypeDef *hfdcanx, uint8_t id, int16_t pos, int16_t vel, int16_t tqe)；
```

**Note:**

1. This mode limits the output torque. If the maximum torque is set too low, the motor may not be able to reach the target velocity.

### Position, Velocity, and Acceleration Mode (Trapezoidal Control)

**Description:**

1. The motor moves with a constant acceleration, achieving a trapezoidal velocity profile: accelerate → constant speed → decelerate, and requests the motor to return status information.
2. Underlying implementation:

```c
void motor_control_pos_vel_acc(FDCAN_HandleTypeDef *hfdcanx, uint8_t id, int16_t pos, int16_t vel, int16_t acc)；
```

**Note:**

1. Supported starting from motor firmware v4.6.0.

### Motion Control Mode (MIT Control)

**Description:**

1. The output torque is calculated as:

```text
Output torque = position error * kp + velocity error * kd + torque.
```

2. Mode usage notes:
    1. When kp=0 and kd=0, specifying tqe achieves a fixed torque output. In this case, the motor will continuously output a constant torque.
    2. When kp=0 and kd≠0, specifying vel achieves constant-speed rotation. A steady-state velocity error exists during constant-speed rotation; additionally, kd should not be too large, as excessively large kd can cause oscillation.
    3. When kp≠0 and kd=0, oscillation will occur. That is, when controlling position, kd must not be set to 0, otherwise the motor will oscillate and may even become uncontrollable.
    4. When kp≠0 and kd≠0, there are multiple scenarios; two common cases are briefly described below:
        - When the target position pos is constant and the target velocity vel is 0, fixed-point control is achieved. The system will drive the actual position toward pos and the actual velocity toward 0, eventually coming to rest at the target position.
        - When pos is a continuously differentiable function of time and vel is the derivative of pos, position tracking and velocity tracking are achieved, i.e., rotating by the desired angle at the desired velocity.
3. Underlying implementation:

```c
void motor_control_pos_vel_tqe_kp_kd(FDCAN_HandleTypeDef *hfdcanx, uint8_t id, int16_t pos, int16_t vel, int16_t tqe, int16_t kp, int16_t kd)；
```

**Note:**

1. Appropriate values of `kp` and `kd` must be set, otherwise the control performance may be poor.
2. Supported starting from motor firmware v4.6.0.

### Stop Mode

**Description:**

1. The motor enters stop mode; all three phases are left floating, allowing the motor to rotate freely.
2. Underlying implementation:

```c
void set_motor_stop(FDCAN_HandleTypeDef *fdcanHandle, uint8_t id)；
```

### Brake Mode

**Description:**

1. All motor phases are shorted to ground, achieving a "damping brake" effect.
2. Braking resistance is positively correlated with motor speed.
3. Underlying implementation:

```c
void set_motor_brake(FDCAN_HandleTypeDef *fdcanHandle, uint8_t id)；
```
